## [Unreleased]

### Added

- Capture ingest for pcap, pcapng, and gzip over raw 802.11, radiotap, Prism, AVS, PPI, and Linux cooked link layers, with tiered FCS recovery (ported from the WPAWolf front-end).
- Per-BSSID encryption classification (WEP / WPA / open) from the Beacon Privacy bit, RSN and WPA information elements, and the data-frame Extended-IV bit; matches aircrack-ng's determination.
- WEP frame harvesting (FR-WEP): IV / Key-ID / ICV extraction, IV+keystream samples from the known LLC/SNAP header and ARP plaintext, the per-Key-ID context, and Shared-Key-authentication keystream recovery.
- Attacks: PTW (Klein/sigma vote with a bounded candidate search), KoreK (17 weighted RC4 key-schedule correlations), FMS, the Sepehrdad RC4-bias database (see below), dictionary (raw and hex), weak-key generators (Neesus-Datacom 40-bit and MD5 104-bit), and a gated 40-bit brute force. Every candidate key is confirmed by RC4-decrypting real frames and checking the CRC-32 ICV -- the only acceptance path.
- Optimal `a_opt` bias weighting: the sparse decisive KoreK correlations (u15/s13/u13_*/u5_2/...) are weighted by the FSE-2013 optimal coefficient `a_opt = (p-1/N)/[(p-1/N)+(2/N)(1-1/N)(1-q/N)]` computed from their measured (p,q), so they carry influence proportional to how decisive they are when they fire (p ~ 20-50/N) rather than a flat per-attack weight; the ever-present Klein/Maitra-Paul stay at their established weight because lowering the high-density workhorse degrades argmax accumulation.
- Polya / global-margin adaptive search: when the fast paths fail, search depth is allocated per key octet by vote uncertainty under a key-test budget (aircrack-ng's global candidate expansion / the Smashing-WEP Polya rank model) -- confident octets stay at their argmax, while flat octets (a strong key byte whose Klein bias vanishes, or an unknown IPv4-ID octet) get many candidates -- so strong bytes and unknown octets are handled without special-casing, adapting the search shape to each capture instead of fixed rungs.
- Two-keystream IPv4 voting: the unpredictable IPv4 Don't-Fragment octet is voted both ways (DF-set and DF-clear, weighted by the observed 220/36 split, mirroring aircrack-ng's `known_clear`), so the key octet it feeds votes correctly whatever the frame's DF state; the larger unknown IPv4-ID octets are handled by the adaptive search above.
- Destination-driven known-plaintext (FR-WEP-6): the 802.11 Destination Address stays in the clear even when the MSDU is WEP-encrypted, and multicast/broadcast group MACs map deterministically to an L3 protocol (RFC 1112 / RFC 2464 / IEEE 802.1X), so the right known plaintext is chosen per frame without decrypting -- more precise than guessing by length. The headline win is IPv6 Neighbor Discovery (a `33:33:..` destination): IPv6 has no header checksum and ND mandates a hop limit of 255, so SNAP + EtherType `86DD` + the IPv6 fixed-header start give sixteen reliable octets (WEP-104 reach) on frames the IPv4-shaped guess mis-keyed from the EtherType on -- pure noise before. EAPOL (the PAE group address) contributes the correct `888E` SNAP prefix, and IPv4 multicast/broadcast (mDNS `224.0.0.251`, SSDP `239.255.255.250`, IGMP `224.0.0.x`, DHCP to `255.255.255.255`) are confirmed IPv4 and mined with the IPv4 header rather than a length guess. aircrack-ng's `known_clear` keys off frame contents it cannot see when encrypted; keying off the cleartext L2 destination is strictly more information. (Honest reach limit: the IPv4 app-layer past the IPv4 header sits behind the ID/checksum/source-address gaps, which the WEP cumulative-sum key chain cannot bridge, so those protocols extend classification precision rather than contiguous keystream; IPv4 unicast ICMP carries no cleartext signal and is not mined -- ICMPv6 is covered by the IPv6-ND path.)
- Per-key-slot cracking (FR-ATK-SLOT-1): a WEP AP can run up to four keys at once (Key ID 0-3), whose frames use different key schedules. aircrack-ng keys its vote table by BSSID alone, so a busy slot drowns the others and at most one key is reported. WEPWolf tags every sample and verifier frame with its Key ID and, for a multi-slot AP, attacks each slot from only its own samples -- recovering and reporting a verified key per slot. The common single-slot capture is unchanged.
- Unique-IV feasibility gating (FR-OUT-5): the statistical attacks tally each distinct IV once, so feasibility now gates on the unique-IV count, not the raw frame count -- a capture that replays one packet is frame-rich but IV-poor and can never converge, and attacking it only burned the per-BSSID budget before. This matches how aircrack-ng decides "enough IVs", and it splits the uncracked networks into genuinely-thin (below the floor) versus material-present-but-no-key.
- Richer uncracked diagnostics (FR-DEBUG-3): `--debug` now reports, per uncracked WEP BSSID, the unique-IV count versus raw frames, which key lengths cleared the feasibility floor, how far a thin capture falls short, and the likely cause when material was sufficient (multiple key slots in use, sparse known-plaintext, or below the practical packet count) -- turning "too thin" into something actionable.
- The full Sepehrdad "Smashing WEP" bias database (the `bias` attack): the complete Table 1 from [Sepehrdad-Vaudenay-Vuagnoux, FSE 2013] -- Klein-Improved and Maitra-Paul, the KoreK A_* correlations, the four negative attacks, and SVV_10 -- each voting for the same cumulative key sum in the key-independent (IV-only KSA) frame, weighted by its published bias strength, into one table resolved by the shared PTW search. This is the most efficient passive WEP attack published and the one aircrack-ng does not ship; voting many biases together recovers a WEP-104 key from a marginal capture (~30k packets in the test) where the Klein-only signal aircrack-ng's PTW uses does not. Every bias is unit-validated to predict the true key sum above the 1/256 chance floor (and the negatives below it); the aircrack-only A_u5_4 is omitted as it is not in Table 1 and measured as anti-signal in this frame. The raw SAC-2010 black-box relations (New_bb_*) are likewise excluded after evaluation -- they are seed-relative, and in WEP the leading seed octets are the known IV, so they carry no secret-key signal (measured at the 1/256 floor); shipping them would be noise. WEP-232's high key octets are recovered by the sequential KoreK attack rather than this parallel bias frame, whose key-independent vote decays with key position -- the database is unit-validated to vote the low WEP-232 octets correctly, and the ell=16 key-repetition merge is out of scope (the 16-octet intermediate key is rejected by C6).
- WEP-104 PTW from ordinary traffic: known plaintext is reconstructed from the predictable start of the encapsulated packet -- ARP's fixed header (also detecting the padded 54-octet MSDU), or the IPv4 header whose total-length field is read from the captured frame size -- giving the fifteen keystream octets WEP-104 needs where the seven-octet SNAP prefix falls short. PTW pins the strong leading octets and sweeps the weakest trailing octet exhaustively. Mirrors aircrack-ng's `known_clear` and last-keybyte bruteforce; WEPWolf now recovers every WEP key aircrack-ng does on the captures it has been validated against.
- Maitra-Paul second vote in PTW: each packet now casts two independent estimates of every key octet's cumulative sum -- the Klein vote from keystream octet `2+i` and the Maitra-Paul vote from the next octet `3+i` -- weighted by their respective biases (13:4, from p ~ 1.36/N and ~ 1.11/N) into the same table. aircrack-ng's PTW votes Klein alone, so the extra per-packet signal lowers the packet count needed to rank every key octet correctly (measured: ~10% fewer mis-ranked octets at a marginal count). Per [Sepehrdad-Vaudenay-Vuagnoux, FSE 2013] Table 1 and [Sepehrdad et al., SAC 2010] Eq. 16.
- Two-pass per-BSSID attack scheduling: each network is swept with a quick pass that runs only the cheap argmax/top-k search of every attack across every key length, then a full pass that adds the expensive backtracking ladders. This stops a wrong key-length's heavy search from consuming the whole per-BSSID time budget before the true length's cheap crack runs -- without it, a real WEP-104 capture with ~100k IVs (recovered in under a second once its length is tried) was left uncracked because the WEP-40 hypothesis's ladder exhausted the budget first. Found by differential validation against aircrack-ng on real captures.
- Adaptive margin-ranked PTW search: when the fast paths fail, every key position is ranked by how decisive its vote is, and a ladder of `(positions, depth)` rungs searches the least-confident positions more deeply -- the `(2, 256)` rung sweeps the pair of fully-unknown octets that real passive IP traffic leaves (the IPv4 Identification field is unpredictable, so its two interior key octets vote pure noise and the old top-k backtrack + trailing brute both miss them), and the wider, shallower rungs reach the several moderately mis-ranked octets a marginal capture leaves. This lowered the clean-ARP WEP-104 recovery count from ~80k packets to ~40-50k (into aircrack-ng's range) and recovers the real-IP-traffic case the old search could not. Each rung stays under a node budget and the per-BSSID deadline still bounds wall-clock; only reached when the clean paths fail, so an ordinary crack is never slowed. Generalises aircrack-ng's candidate expansion and its brute force of the unrecoverable IP-ID sigmas, bounded for the passive offline setting.
- Parallel multi-file ingest (FR-IN-6): the input files are parsed concurrently on the work-stealing pool instead of one after another -- each file is streamed into its own per-BSSID map and accounting with no shared mutable state on the hot path, then the per-file maps are merged in input-file order. The merge is order-fixed, so the result (per-BSSID material and every banner counter) is byte-for-byte identical regardless of thread scheduling or `--threads` -- verified by a determinism test. A large set of capture files now ingests in a fraction of the wall-clock; aircrack-ng reads its inputs serially. Files are ingested in bounded batches that fold into the result and are freed before the next, so peak memory does not grow with the file count. The opt-in `--log` and `--carve` sinks stay correct under concurrency: log events are buffered per file and replayed in order so `file=` attribution holds, and the shared carve writer is locked only per written frame (never on the common path).
- Distinct-IV voting: WEP reuses its 24-bit IV heavily, so the statistical attacks tally each distinct IV once (aircrack-ng's unique-IV table), keeping reused IVs from biasing the votes.
- aircrack-style search tuning: `-f`/`--fudge` (the adaptive KoreK/bias ratio cut -- keep every candidate voting at least `top / factor`), `-x`/`--bruteforce` (exhaustive last-keybyte sweep), and `-c`/`--alnum` (restrict candidate octets to printable ASCII).
- SIMD-accelerated ICV verification: the CRC-32 fold runs on PCLMULQDQ via the audited `crc32fast` crate, and RC4 frame decryption folds the keystream in 16 octets at a time with a hand-written SSE2 XOR kernel. `unsafe_code` is `deny` (not `forbid`), permitted only behind reasoned `#[allow]` in `src/simd.rs` and gated by the byte-exact-equals-scalar harness.
- Batched-RC4 brute kernel (FR-SIMD-2): the 40-bit grind now runs a known-plaintext prefilter instead of a full decrypt-and-verify per key. A batched RC4 kernel key-schedules a whole candidate-key batch with the lanes interleaved -- each lane's S-box is independent, so the CPU overlaps the data-dependent gather/scatter latency that bounds a single scalar stream (cross-lane ILP) -- and produces four keystream octets per key. Because the WEP MSDU starts with the LLC/SNAP header (`AA AA 03 00`), a wrong key clears that four-octet filter with probability 2^-32, so over the 2^40 space only ~2^8 candidates ever reach the full `Verifier` (which stays the sole acceptance path, C4 -- the prefilter only rejects). End-to-end this is ~4x the scalar grind on the benchmark (0.5M -> 2.2M keys/s); it is the same known-plaintext shortcut aircrack-ng's brute uses (C5). The kernel is byte-exact with the scalar oracle on every SIMD tier (the FR-SIMD-2 gate); vectorising the S-box itself was measured and rejected (x86 `vpgatherdd`/`vpscatterdd` retire one element per micro-op, no faster than the interleaved scalar lanes), so all tiers share the one portable kernel and the detected tier only labels the backend.
- Robustness on real captures: a per-key-length feasibility gate skips statistical attacks on captures too thin to crack (FR-OUT-5), and a per-BSSID wall-clock deadline (default 30s, `--time-budget` overrides) keeps a hard network from starving the run -- validated on a large multi-file input of tens of millions of packets (cracked in a few seconds, no spin), with peak memory bounded and proportional to the WEP material rather than the file count or sizes.
- A hashcat-style potfile (`--potfile`): recovered keys are appended as `bssid:key_hex` and an existing potfile seeds reuse, so a known network is reported without re-attacking it.
- WEP-focused output: the table lists WEP BSSIDs most-IVs-first with a one-line census of the rest (not every observed network), and ESSIDs are sanitised so binary bytes cannot mangle the terminal.
- Richer `--debug`: per-BSSID material (IV/ARP/SKA counts) and, per WEP BSSID, why it was not cracked -- too thin, or enough material but no single recoverable key (FR-DEBUG-2/3).
- Sweep/grind scheduler: the cheap attacks run BSSID-parallel on a work-stealing pool (the sweep), a key recovered on one BSSID is reused against co-located same-key access points, and the 40-bit brute runs one BSSID at a time on the full pool (the grind) with a shared cancel flag, so a hit or an exhausted `--time-budget` abandons in-flight work within microseconds.
- RC4 and IEEE CRC-32 primitives with a runtime SIMD-tier detector, proven byte-exact against the canonical published vectors.
- Three output surfaces, each carrying the same three sections -- the recovered keys, the WEP-BSSID summary (most IVs first), then the stats breakdown: a column-aligned human table (default), `--plain` (tab-separated records tagged `key` / `wep` / `stat`), and `--json` (typed NDJSON: a `key` then `bssid` object per row, then one `stats` object). Every surface emits the full per-key record (bssid, essid, key hex/ascii/bits, key id, attack, unique IVs, crack seconds); `--quiet` reduces any surface to the keys. The stats banner groups its per-packet and per-BSSID reconciliation into ingest / networks / run sections, accounts for every read packet by frame class (with drop causes broken out when nonzero), attributes each key to its attack, and reports the sweep / grind timing and peak RSS, closed by a `wepwolf <version> (<git>)` footer.
- Targeting and tuning flags: `-b`/`--bssid` (one network), `-n`/`--keylen` (restrict to 40/104/232-bit hypotheses), `-j`/`--threads` (size the sweep pool), and `--time-budget` (per-BSSID brute cap). Short options follow aircrack-ng (`-b`/`-n`/`-f`/`-x`/`-c`/`-q`/`-w`) where they correspond.
- Live stderr progress: an ingest spinner, a sweep bar showing BSSIDs done with elapsed / ETA and the network currently being attacked (each recovered key streamed above it as it verifies), and a brute keyspace bar naming the network being ground with percent / keys-per-second / ETA / active SIMD tier. Shown only on a terminal with the default table output; suppressed under `--plain`, `--json`, `--quiet`, and `--debug`.
- Frame carving (`--carve FILE`): write every parsed WEP frame (WEP-encrypted data and Shared-Key auth) plus each WEP network's beacon/probe into a standalone pcap -- raw 802.11 with zeroed timestamps, streamed within bounded memory. Because wepwolf's own parser (with its tiered link-header recovery) selects and writes the frames, the output is exactly the WEP set it cracks from -- something a generic dissector cannot reproduce on malformed headers -- and the mixed radiotap/Prism/AVS link layers of the inputs all normalise to one link type that both wepwolf and aircrack-ng read. A multi-file capture set collapses into one self-contained, re-crackable capture.
- Diagnostics: `-d`/`--debug` (timestamped stderr), a categorized `-l`/`--log FILE`, and an RSS memory monitor with a peak-RSS banner row.
- A spec-driven development system (`specs/`) with machine-checked requirement-to-test coverage (`make audit`) and a diagnostics-banner contract (`make audit-stats`).

### Notes

- WEPWolf is passive and offline: it reads capture files only, and never captures traffic, injects frames, or touches a radio.
- On the aircrack-ng `wep_64_ptw.cap` reference capture it recovers the key in roughly a fifth of a second.
